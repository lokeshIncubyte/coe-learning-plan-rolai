# exec-cycle

## Inputs

- `<project>/cycles/NNN-<kebab-slug>.md` — next pending cycle (lowest numeric prefix with `status: pending` or no status field). Cycle files use three-digit zero-padded prefixes (`001`, `002`, …) set by `plan-cycle`.
- `<project>/docs/spec/checklist.md` — to tick items when all sibling cycles for that item are done

> This skill reads cycle files generated by `plan-cycle`. For projects using a single `tdd-plan.md`, use `tdd-cycle` instead.

## Output

One squash commit on `main`, branch deleted locally, cycle file marked `status: done`, checklist item ticked if all siblings are done.

### Squash commit format

```
feat(cycle-NNN): <one-line unit under test>

RED: <test-file> — <what the assertion checked>
GREEN: <files created/modified> — <minimum change that made RED pass>
REFACTOR: <what was improved, or "none">
```

The body re-states the narrative erased by squash so that `git log` remains a readable TDD history.

## Rules

- **Test-first.** No production code without a failing test demanding it.
- **Smallest GREEN.** Implement only what the cycle's GREEN section specifies.
- **One commit per phase** (red / green / refactor) on the feature branch; `main` gets one squash commit.
- **`main` always stays green** between invocations.
- **Partial branch guard.** If `tdd/<slug>` already exists locally → stop, tell the user, do not re-start.

## Steps

- [ ] 1. Pick next cycle (NO STOP)
- [ ] 2. Branch from `main` (STOP — WAIT)
- [ ] 3. Write failing test (NO STOP)
- [ ] 4. RED proof (NO STOP)
- [ ] 5. RED commit (NO STOP)
- [ ] 6. GREEN implementation (NO STOP)
- [ ] 7. GREEN proof (NO STOP)
- [ ] 8. Review GREEN diff and commit (STOP — WAIT)
- [ ] 9. REFACTOR — skip to 12 if "none" (NO STOP)
- [ ] 10. REFACTOR proof (NO STOP)
- [ ] 11. Review REFACTOR diff and commit (STOP — WAIT)
- [ ] 12. Squash-merge to `main` (STOP — WAIT)
- [ ] 13. Mark done and tick checklist (NO STOP)

---

### 1. Pick next cycle (NO STOP)

Scan `cycles/` for all `.md` files. Find the file with the lowest numeric prefix where frontmatter `status` is `pending` or absent. Announce: cycle id, slug, and the Behavior paragraph.

**If `tdd/<slug>` already exists locally** → stop. Tell the user partial work remains on that branch; do not re-start.

### 2. Branch from `main` (STOP — WAIT)

```bash
git checkout main
git pull
git checkout -b tdd/<slug>
```

### 3. Write the failing test (NO STOP)

Write exactly the assertion from the cycle's RED section into the specified test file. No production code.

### 4. RED proof (NO STOP)

**Expected:** test fails.

**If the test passes:** the unit already exists.
- Strengthen the RED assertion in the cycle file to target a deeper missing behavior, or
- Set `status: skip` in the cycle frontmatter and move to the next cycle.

In either case, stop and tell the user before continuing.

```bash
<pm> test -- <test-file>
```

### 5. RED commit (NO STOP)

```bash
git status          # verify only <test-file> is staged; stop if unexpected files appear
git add <test-file>
git diff --cached   # confirm exactly the test file
git commit -m "red: <slug>"
```

### 6. GREEN — smallest production code (NO STOP)

Implement only what the cycle's GREEN section specifies. Run framework generators where applicable.

### 7. GREEN proof (NO STOP)

```bash
<pm> test -- <test-file>
<pm> test
```

Both must pass. If a previously passing test now fails → revert and try a smaller change.

### 8. Review GREEN diff and commit (STOP — WAIT)

Surface `git diff` and a one-line summary. Check: smallest change? scope creep? naming OK? Adjust if needed, re-run step 7. When good:

```bash
git add <files>
git status
git diff --cached
git commit -m "green: <slug>"
```

### 9. REFACTOR — planned changes only (NO STOP)

**Precondition:** if the cycle's REFACTOR section is "none" → skip steps 9–11 and go directly to step 12.

Apply only the refactors listed in the cycle's REFACTOR section.

### 10. REFACTOR proof (NO STOP)

```bash
<pm> test
```

Full suite must pass. If anything turns red → refactor changed behavior; revert and treat as a new cycle.

### 11. Review REFACTOR diff and commit (STOP — WAIT)

Surface `git diff` and one-line summary. Check: readability improved? no behavior sneak-in? Adjust if needed, re-run step 10. When good:

```bash
git add <files>
git diff --cached
git commit -m "refactor: <what changed>"
```

### 12. Squash-merge to `main` (STOP — WAIT)

Verify before approving:
- Working tree is clean (`git status` shows nothing uncommitted)
- All phase commits appear on the branch (`git log --oneline tdd/<slug>`)
- Full suite is green

```bash
git status
git checkout main
git pull
git merge --squash tdd/<slug>
git commit -m "$(cat <<'EOF'
feat(<id>): <one-line unit under test>

RED: <test-file> — <what the assertion checked>
GREEN: <files created/modified> — <minimum change that made RED pass>
REFACTOR: <what was improved, or "none">
EOF
)"
git branch -D tdd/<slug>
```

### 13. Mark done and tick (NO STOP)

Set `status: done` in the cycle file frontmatter.

**Tick logic:**
1. If the cycle has a `group` field, check all `cycles/` files sharing that group. If any sibling is still `status: pending` → print "Checklist tick deferred — <sibling-id> still pending." and stop here.
2. If all siblings are done (or the cycle is atomic with no `group`), attempt tick-back:
   - Read the cycle's `source` field.
   - If `<project>/docs/spec/checklist.md` exists and contains a line matching `source` → change `- [ ] <source>` to `- [x] <source>`.
   - If no matching checklist line is found → skip tick silently. The `source` field may be a feature description or list item with no corresponding checklist entry; this is expected.
